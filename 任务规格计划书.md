# Claw 插件混淆代码恢复：任务规格计划书（可维护开发版）

## 1. 背景与目标

本仓库当前应视为“可运行发行产物 + 人工回填可维护层”的半恢复仓库，而不是可直接重建的原始源码工程。

本计划的最终目标不是 1:1 还原原作者的 TS/React/Vite 目录，而是：

- 在不破坏现有运行行为的前提下，让关键逻辑逐步变得可读、可测、可维护
- 让后续新需求优先落在可维护层，尽量减少直接改哈希 bundle 的次数

## 2. 当前现状（截至 2026-04-14）

已确认：

- 根目录缺失原始构建骨架，核心界面仍由 `manifest.json` 加载 `assets/*` 哈希 bundle 驱动。
- `DEOBFUSCATION_MAP.md` 用于记录“自动反混淆 + 人工入口标注”的进度和锚点。
- 已存在可维护层（如 service worker runtime、custom provider、github update 等），并有回归测试可跑。

本轮已完成的工作类型：

- 对关键 bundle 做“定点语义拆解”：只加注释与 `const` 语义锚点/别名，不做大规模清洗
- 添加锚点回归测试：保证锚点文本可被 `rg` 搜索且不被后续改动破坏

## 3. 不可变契约（第一优先级）

第一阶段必须冻结并保护这些隐式契约，避免破坏 bundle 与壳层之间的约定：

- `manifest.json` 的入口文件、HTML 页面、权限声明、资源路径不变
- `chrome.storage` 关键键名不变（包括前缀、scope 结构等）
- `chrome.runtime.sendMessage` / `onMessage` 的 action/type 名不变
- sidepanel/options/pairing 等页面的脚本加载顺序不变
- 已存在的 `globalThis.__CP_*` 命名空间不改名、不反向依赖新构建体系

稳定契约统一收口在 `claw-contract.js`，新增契约只能“增量追加”，不得破坏既有字段语义。

## 4. 工作方法与约束

### 4.1 对混淆 bundle 的改动规则

允许：

- 增加中文语义注释（用于标记关键入口/分支）
- 增加 `const __cp* = <原变量/函数>` 形式的语义锚点/别名
- 在不改变语义的前提下，用 `const` 替换“硬编码字符串”引用（仅当不会影响运行）

禁止：

- 大规模重排/格式化 bundle
- 替换算法/逻辑、修改返回值、改变异步时序
- 引入需要构建的 TypeScript/打包链作为运行依赖

### 4.2 对可维护层的改动规则

优先把“可变需求”写在根目录可维护模块中，通过：

- `service-worker-loader.js` 的加载顺序
- `globalThis.__CP_*` 运行时挂载点
- storage/message 的同接口替换

来覆盖/扩展 bundle 行为，而不是直接动 bundle。

## 6. 任务拆解（用于子代理并行推进）

本仓库适合“多文件独立处理”的并行模式。每个子任务必须有明确写集，避免写冲突。

推荐的工单模板：

- 目标：一句话说明要恢复/外提的能力
- 文件边界：只允许修改哪些文件
- 允许操作：只加注释/锚点/别名，或允许外提模块
- 输出：必须给出可 `rg` 搜索的锚点文本清单 + 变更文件列表
- 验收：新增/更新哪些测试

## 8. 风险与回滚策略

主要风险：

- bundle 最小改动仍可能影响运行（拼写错误、插入位置破坏作用域）
- 键名/消息名漂移导致“静默失效”（最难排查）

回滚策略：

- 所有对 bundle 的改动都必须伴随锚点回归测试
- 如出现行为异常，优先回滚最近一次 bundle 变更，再用更小粒度锚点重做

第三轮 permission sentinel 补齐（2026-04-15）：

- `handleToolCall(...)` 返回带 `type` 的对象时，当前约定它仍是 `permission_required sentinel`，不是最终 `tool_result`

- `actionData` 只回填 permission popup 预览所需的最小字段，不是完整执行上下文
- `javascript_tool`：只透出脚本文本，不带 timeout / eval 细节
- `form_input / upload_image`：只回填 `ref/coordinate/value/imageId` 这类最小定位字段
- `find / get_page_text / read_page / read_console_messages / read_network_requests / navigate`：默认只返回基础四元组，不附 `actionData`
- `GIF 导出上传`：复用 `upload_image` 家族，但 popup 只吃 `coordinate` 预览，不携带 gif 二进制
- `navigate`：只有普通 URL 分支会做协议补全、blocklist 与域 permission；`back/forward` 不重走 permission gate
- `get_page_text / read_page`：共用“先 permission，后 hideIndicator/read DOM”的读页模板
- `upload_image`：先过目标域 permission，再做原始 URL 安全校验与 message image 解析
- `A(...) / __cpMcpPreExecuteSameDomainGuard`：域权限放行后的同域重校验闸门，覆盖 `click/type/key/drag/scroll_to/hover/javascript_tool/file upload/form_input/GIF export upload/upload_image`
- `A(...)` 的调用家族：
  - 页面级动作：`type / key / javascript_tool`
  - 定位/指针动作：`click / double_click / triple_click / drag / scroll_to / hover`
  - 表单/上传动作：`file upload / form_input / upload_image / GIF export upload`
- `M / __cpMcpScreenshotViewportContextLedger`：screenshot viewport context 账本，只保存尺寸字段给坐标换算
- `ne(...) / __cpMcpScaleScreenshotCoordinatesToViewport`：截图坐标 -> 当前 viewport 坐标缩放器
- `me(...) / __cpMcpResolveRefToViewportCenter`：`read_page/find ref -> 可交互中心点坐标` 解析器，会清理失效 ref、`scrollIntoView(center)`，并写 `ref_lookup_ms`
- `window.__generateAccessibilityTree`：`read_page / find` 共享的 ref writer；先复用旧 ref，再分配新 ref_X，并在生成结束后 sweep 一次陈旧 WeakRef
- `file_upload / form_input / upload_image(ref)`：都是直接 ref consumer，不走 `me(...)`；它们会在各自页面注入函数里直接验活 `__claudeElementMap`
- `file_upload`：只有命中 `<input type=file>` 才继续桥接到 `DOM.setFileInputFiles`
- `upload_image(ref)`：命中 `<input type=file>` 走 files 注入，否则统一按 drag-drop target 派发事件，不做 droppable 能力校验
- `clearContext / clearAllContexts`：当前只见定义，没有显式调用；screenshot context 主要靠后续 `setContext(...)` 覆盖刷新
- `scroll`：会复用 screenshot context 做坐标缩放，但不像 `click / hover / drag` 那样再进 `A(...)`
- `zoom`：只消费最近一次 screenshot context，不会回写或清理账本
- `click(ref)` / `hover(ref)`：先解 `ref`，再进 `A(...)`
- `scroll_to`：先过 `A(...)`，再解 `ref / scrollIntoView`
- `click(coordinate)` / `hover(coordinate)` / `drag`：先按截图上下文做坐标缩放，再进 `A(...)`
- `Me(...) / __cpMcpBeforeunloadNavigationGuard`：`navigate` 专用 beforeunload 三态收口器
- `update_plan`：审批通过后直接回 `plan approved` 文本，不重跑普通工具
- `rn(...) / __cpDomainTransitionPermissionRequiredFactory`：`DOMAIN_TRANSITION` 专用 producer，本地生成 `toolUseId`

- `OPEN_SIDE_PANEL`：`tabId` 只负责打开/绑定目标 sidepanel；后续 `POPULATE_INPUT_TEXT` 不再携带 `tabId`

- `POPULATE_INPUT_TEXT`：自身不按 `tabId/sessionId` 过滤；选路发生在 `OPEN_SIDE_PANEL / EXECUTE_TASK`
- `ACK 回包收口`：只消费 `success`；`secondaryTabId/mainTabId/timestamp` 只服务探测阶段与缓存
- `MCP_PERMISSION_RESPONSE`：service-worker 主桥只做轻量 ACK，不解析 `requestId/allowed`
