# Claw in Chrome 学习路线

## 1. 文档目的

这份文档不是泛泛的“浏览器插件学习计划”，而是针对当前仓库的实际结构制定的一份可执行学习清单。

这个项目的特点很鲜明：

- 它是一个 Chrome Manifest V3 扩展
- 仓库里同时存在发行 bundle 和后续人工补回的可维护层
- 真正适合学习和二次开发的入口，主要在根目录的可维护脚本中
- `assets/*.js` 中的大 bundle 更适合在你建立全局认知后做对照阅读，而不是一开始就硬啃

如果你按这份路线走完，目标不是“把每个文件都背下来”，而是达到下面这几个结果：

- 你能说清这个扩展是怎么启动的
- 你能说清 sidepanel、options、service worker、offscreen 的关系
- 你能独立追踪一条消息链路和一条存储链路
- 你能修改一个小功能，并知道改哪里风险最低
- 你能逐步进入 `assets/*.js` 中做有边界的深入分析

## 2. 学习总原则

### 2.1 先学可维护层，再碰 bundle

这个仓库最容易让人踩坑的地方，就是一上来就钻进：

- `assets/sidepanel-BoLm9pmH.js`
- `assets/options-Hyb_OzME.js`
- `assets/service-worker.ts-H0DVM1LS.js`
- `assets/mcpPermissions-qqAoJjJ8.js`

这些文件不是不能学，而是不适合当第一站。

更合理的顺序是：

1. 先看入口和契约
2. 再看根目录的可维护运行时
3. 再看 provider 体系
4. 最后再回看 bundle，对照行为去理解

### 2.2 读代码时始终回答 4 个问题

每读完一个文件，你都要尝试回答：

1. 这个文件的入口在哪里
2. 它依赖了哪些 `globalThis.__CP_*`、`chrome.*`、storage key 或消息类型
3. 它修改了哪些状态
4. 它在整个系统中的位置是什么

### 2.3 不求一次读懂，先建立地图

第一次阅读的目标不是理解每一行，而是建立“地图感”。

推荐你先把一个模块看成：

- 入口
- 核心职责
- 对外接口
- 关键副作用

等你第二轮再深入看细节。

## 3. 项目整体认知

### 3.1 先记住这条主链

```text
manifest.json
  -> service-worker-loader.js
  -> service-worker runtime / 其他可维护层
  -> sidepanel.html / options.html / offscreen.html / pairing.html
  -> 各页面自己的 bundle + 根目录增强脚本
```

### 3.2 先记住这几个模块分层

- 扩展入口层：`manifest.json`
- 后台层：`service-worker-loader.js`、`service-worker-runtime.js`
- 独立窗口与会话补偿层：`service-worker-detached-window-runtime.js`
- 自定义模型层：`custom-provider-models.js`、`provider-format-adapter.js`
- 设置页增强层：`custom-provider-settings.js`、`options-update-enhancer.js`
- 侧栏内联配置层：`sidepanel-inline-provider.js`
- 更新能力：`github-update-shared.js`、`github-update-worker-runtime.js`
- 黑盒大 bundle：`assets/*.js`

### 3.3 你的第一认知结论应该是

这个仓库不是“前端页面工程”，而是“已打包扩展 + 逐步恢复可维护边界”的工程。

这句话很重要。

只要你脑子里一直把它当成一个常规 `src/` 项目，你就会不断迷路。

## 4. 学习前准备清单

在正式开始前，先完成下面这些准备动作。

### 4.1 环境准备

- 安装 Chrome 或 Edge
- 打开 `chrome://extensions/`
- 开启开发者模式
- 加载当前仓库目录为已解压扩展
- 先手动点开一次 sidepanel
- 先手动打开一次 options 页面

### 4.2 命令准备

建议先熟悉这几个命令：

```bash
npm run test:unit
npm run test:integration
npm run test:e2e
npm test
rg -n "关键字" 文件名
rg --files
```

### 4.3 学习记录准备

建议你单独准备一个笔记文件，名字随意，比如：

- `我的学习笔记.md`
- `trace-notes.md`

每次记录只写 4 项：

- 文件名
- 入口
- 核心职责
- 我当前还没搞懂的点

### Day 4：看 service worker 装配层

#### 学习清单

- 搞清它为什么既加载 bundle，又加载根目录 runtime
- 搞清 detached window runtime 是如何被注册进 service worker runtime 的

#### 你当天要输出的结果

- 一张 service worker 启动链路图
- 一段你自己的总结：为什么这个 loader 文件很重要

#### 指导意见

- 不要嫌这个文件短，它是整条后台启动链的“装配总线”
- 读这类文件时，重点不是逻辑量，而是依赖关系

### Day 5：回顾与复盘

#### 复盘清单

- 如果这天你发现自己还说不清全局结构，不要急着进入第 2 周

## 10. 每周固定复盘清单

每周结束后，都建议你做一次复盘。

### 复盘问题

1. 这周我新增理解了哪几条主链路
2. 哪些文件我能说清职责，哪些还只能模糊描述
3. 哪几个 storage key / message type 我已经熟了

## 11. 实操建议

### 11.1 阅读顺序建议

推荐固定使用下面的阅读动作：

1. 先读文件顶部常量
2. 再读导出对象或注册逻辑
3. 再读主流程函数
4. 最后读工具函数和边界分支

### 11.2 追踪问题建议

遇到一个行为时，优先这样追：

1. 谁触发了它
2. 它发了什么消息
3. 后台或页面谁收了这个消息
4. 它最终写了什么 storage
5. UI 最后怎么变化

### 11.3 做笔记建议

不要摘抄代码。

更推荐你写：

- “这个文件像什么”
- “这个函数是为了解决什么问题”
- “如果删掉这段逻辑会发生什么”

这样的笔记更能帮你真正理解。

### 11.4 调试建议

建议你同时开：

- 扩展页面 DevTools
- service worker DevTools
- 页面本身 DevTools

因为这个项目跨页面、跨上下文很频繁，只看一个控制台很容易缺信息。

## 12. 常见误区

### 误区一：一上来就读大 bundle

后果：

- 细节很多
- 语义很散
- 很难判断哪些是恢复层，哪些是原始逻辑

正确做法：

- 先读根目录可维护层

### 误区二：把所有文件都当成同等优先级

后果：

- 时间被稀释
- 学了很多边缘内容，主线却不清楚

正确做法：

- 优先学主链：manifest -> service worker -> sidepanel/options -> provider

### 误区三：只看 UI，不看 storage 和消息

后果：

- 只能理解表面行为
- 一遇到跨页面问题就断链

正确做法：

- 每次都同时看 UI、消息、storage 三层

### 误区四：一开始就想“完全恢复源码”

后果：

- 心理负担很重
- 容易陷入无效整理

正确做法：

- 先理解可维护边界
- 接受这是一个“半恢复仓库”

## 15. 最后的学习建议

这类仓库最适合的学习方式，不是“背源码”，而是：

- 建地图
- 抓主链
- 追消息
- 追状态
- 做小改动
- 用测试确认理解

你可以把整个过程理解成一句话：

先学系统边界，再学实现细节；先能定位，再求精通。

如果后续你准备继续深入，建议下一步直接做两件事：

1. 自己手动画一版系统架构图
2. 选一个低风险模块做一次最小修改并跑回归

这会比继续被动读代码，成长更快。
